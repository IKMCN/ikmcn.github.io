## What the project asked for

The Math Game spec is deliberately small. Build a console application that:

- Presents the user with random arithmetic questions across the four operations — addition, subtraction, multiplication, division
- Validates input — the user should only be able to enter a number
- Tracks the score for the session
- Lets the user choose a difficulty level
- Loops until the user decides to quit

That's it. No database. No API. No framework. Just C# and a console.

## Why I started here

I've been using AI to generate code for a while. The results look fine on the surface — clean, consistent, well-structured. But when something breaks on site and nobody is there to ask, that knowledge gap becomes very obvious very quickly. I've had that experience with Linux commands and I didn't want to repeat it with C#.

The C# Academy curriculum starts small deliberately. The Math Game teaches the fundamentals — loops, conditionals, methods, input handling — before anything more complex is layered on top. Skipping it because it feels too basic would be missing the point.

## What I built

The application runs a game loop that continues until the user chooses to exit. Each round:

1. The user selects an operation or lets the game pick randomly
2. Two numbers are generated based on the selected difficulty
3. The question is displayed and the user enters an answer
4. Input is validated — non-numeric entries are rejected and the question is repeated
5. The answer is checked and the score is updated
6. At the end of the session the final score is displayed

Division questions only generate numbers that divide cleanly — no decimal answers.

```csharp
static int GetValidInput()
{
    while (true)
    {
        string input = Console.ReadLine();
        if (int.TryParse(input, out int result))
            return result;
        Console.Write("Please enter a number: ");
    }
}
```

That `TryParse` pattern came up immediately — it's the right way to validate numeric input in C# without throwing exceptions on bad input. Understanding *why* that's better than `int.Parse` inside a try/catch is more useful than just having the line.

## What I learned

**Game loops are just while loops.** The structure of a console application with a menu and a loop felt unfamiliar at first but it's straightforward once you see it — a `while (playing)` loop with a menu choice that sets a flag to exit.

**Methods keep things readable.** Breaking the logic into small methods — `GenerateQuestion()`, `GetValidInput()`, `CheckAnswer()` — made the main loop clean and easy to follow. That separation of concerns habit starts here.

**Difficulty affects number range, not operation.** Scaling difficulty by controlling the range of generated numbers (`1-10` for easy, `1-50` for medium, `1-100` for hard) keeps the game fair while feeling progressively harder.

**Integer division needs guarding.** Division questions where the answer is a decimal don't work cleanly in a console game expecting an integer answer. Generating the answer first and working backwards — `answer * divisor = dividend` — guarantees a clean result every time.
