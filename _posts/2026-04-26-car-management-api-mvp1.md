The first instinct when building an API is to jump straight to the controller. Get something running, see a response in the browser, feel like progress is being made. I deliberately did the opposite. MVP 1 is a class library with no API, no database, and no frontend. Just the foundation.

The reasoning is simple — if the foundation is wrong, everything built on top of it is wrong too. Getting the domain model and the layered architecture right before anything else connects to it means the hard decisions are made when they are cheapest to change.

## What the class library contains

The library has four things: a domain model, a repository interface, a repository implementation, and a service layer. Each one has a specific job and talks to the layer below it through an interface, not a concrete class.

```
CarService
    ↓ ICarRepository
CarRepository
    ↓ List<Car> (in-memory for now)
```

Nothing in this stack knows about HTTP, databases, or frontends. That comes later.

## The Car model

The first real decision was what a `Car` looks like. The book this project is based on had most properties typed as strings — `mpg`, `cylinders`, `horsepower` all as `string`. That felt wrong immediately. You cannot sort or filter on a string the way you can on a number, and the types should reflect what the data actually is.

Here is what the model looks like after working through each property:

```csharp
namespace CMS_ClassLibrary.Models;

public enum BodyTypes
{
    Pickup, Saloon, SUV, Hatchback, MPV, Estate, Coupe, Convertible
}

public enum Gearboxes
{
    Manual, Automatic
}

public enum FuelTypes
{
    Petrol, Diesel, Electric, Hydrogen, Hybrid
}

public class Car
{
    public Guid Id { get; init; } = Guid.NewGuid();
    public string Make { get; set; } = null!;
    public string Model { get; set; } = null!;
    public string Colour { get; set; } = null!;
    public int Year { get; set; }
    public decimal Price { get; set; }
    public int Mileage { get; set; }
    public Gearboxes Gearbox { get; set; }
    public BodyTypes BodyType { get; set; }
    public int Doors { get; set; }
    public int Seats { get; set; }
    public decimal EngineSize { get; set; }
    public int BootSpace { get; set; }
    public FuelTypes FuelType { get; set; }
}
```

**The decisions behind the types:**

`Guid Id` with `init` — `Guid` rather than `int` because the system will eventually be distributed across multiple machines. An auto-incrementing int generated by one database does not stay unique across systems. A `Guid` is globally unique by design. The `init` keyword means the Id can be set when the object is created but never changed afterwards — the right constraint for an identifier. `= Guid.NewGuid()` means every new `Car` gets a unique Id automatically without anything having to ask for one.

`decimal` for Price and EngineSize — `Price` needs decimal precision for pence. `EngineSize` needs it for values like `1.4` and `2.0`. An `int` would lose that information.

Enums for `BodyType`, `Gearbox`, and `FuelType` — these are fixed lists that do not change. An enum is the right type for a known, bounded set of values. The compiler enforces valid values so bad data cannot get in.

Strings for Make, Model, and Colour — these were considered for enums. They are not. New manufacturers appear, new models launch every year, and colour names are invented constantly by marketing departments. Maintaining an enum for any of these would be a full time job.

`int` for Doors, Seats, BootSpace, Mileage, and Year — whole numbers, no decimal precision needed.

**What is deliberately missing:**

`IsDeleted` was considered for soft deletes — flagging a car as deleted rather than removing it from the database. It was left out of MVP 1. There is no database yet, so soft deletes have no meaning. It will come back in a later MVP when there is something worth protecting.

## The repository

The repository's job is to handle data storage and retrieval. Nothing else. It does not know about business logic. It does not know about HTTP. It takes a `Car` in or gives a `Car` back and that is the entirety of its concern.

The interface comes first:

```csharp
using CMS_ClassLibrary.Models;

namespace CMS_ClassLibrary.Repository
{
    public interface ICarRepository
    {
        Task<bool> CreateAsync(Car car);
        Task<bool> DeleteByIdAsync(Guid id);
        Task<IEnumerable<Car>> GetAllAsync();
        Task<Car?> GetByIdAsync(Guid id);
        Task<bool> UpdateAsync(Car car);
    }
}
```

Every method is async. This is important — when the repository eventually talks to a real database, those calls will be I/O bound operations that should not block the thread. Building async in from the start means nothing needs to change when the real database arrives in MVP 2a.

`GetByIdAsync` returns `Car?` — the `?` means it can return null. If no car exists with that Id, returning null is the honest answer. Throwing an exception for a missing record is the wrong layer for that decision.

The implementation for MVP 1 uses an in-memory list:

```csharp
using CMS_ClassLibrary.Models;

namespace CMS_ClassLibrary.Repository;

public class CarRepository : ICarRepository
{
    private readonly List<Car> _cars = new();

    public Task<bool> CreateAsync(Car car)
    {
        _cars.Add(car);
        return Task.FromResult(true);
    }

    public Task<Car?> GetByIdAsync(Guid id)
    {
        var car = _cars.SingleOrDefault(x => x.Id == id);
        return Task.FromResult(car);
    }

    public Task<IEnumerable<Car>> GetAllAsync()
    {
        return Task.FromResult(_cars.AsEnumerable());
    }

    public Task<bool> UpdateAsync(Car car)
    {
        var carIndex = _cars.FindIndex(x => x.Id == car.Id);
        if (carIndex == -1)
        {
            return Task.FromResult(false);
        }
        _cars[carIndex] = car;
        return Task.FromResult(true);
    }

    public Task<bool> DeleteByIdAsync(Guid id)
    {
        var removedCount = _cars.RemoveAll(x => x.Id == id);
        var carRemoved = removedCount > 0;
        return Task.FromResult(carRemoved);
    }
}
```

`Task.FromResult()` wraps a synchronous value in a completed Task. It lets the in-memory implementation match the async signature of the interface without actually doing any async work. When `PostgresCarRepository` arrives in MVP 2a, it will use real `await` calls to the database — and nothing else in the system needs to change.

## The service layer

The service sits above the repository. Its job is business logic — deciding what should happen and under what conditions. For MVP 1 the business logic is thin, but the layer exists and the pattern is established.

The interface:

```csharp
using CMS_ClassLibrary.Models;

namespace CMS_ClassLibrary.Services;

public interface ICarService
{
    Task<IEnumerable<Car>> GetAllCars();
    Task<Car?> GetCarById(Guid id);
    Task<bool> CreateCar(Car car);
    Task<bool> UpdateCar(Car car);
    Task<bool> DeleteCar(Guid id);
}
```

The implementation depends on `ICarRepository`, not `CarRepository`:

```csharp
using CMS_ClassLibrary.Models;
using CMS_ClassLibrary.Repository;

namespace CMS_ClassLibrary.Services;

public class CarService : ICarService
{
    private readonly ICarRepository _carRepository;

    public CarService(ICarRepository carRepository)
    {
        _carRepository = carRepository;
    }

    public async Task<IEnumerable<Car>> GetAllCars()
    {
        var cars = await _carRepository.GetAllAsync();
        return cars;
    }

    public async Task<Car?> GetCarById(Guid id)
    {
        var car = await _carRepository.GetByIdAsync(id);
        return car;
    }

    public async Task<bool> CreateCar(Car car)
    {
        var result = await _carRepository.CreateAsync(car);
        return result;
    }

    public async Task<bool> UpdateCar(Car car)
    {
        var result = await _carRepository.UpdateAsync(car);
        return result;
    }

    public async Task<bool> DeleteCar(Guid id)
    {
        var result = await _carRepository.DeleteByIdAsync(id);
        return result;
    }
}
```

`CarService` does not know what `_carRepository` actually is. It knows it implements `ICarRepository`. That is all it needs to know. When the real PostgreSQL repository arrives, `CarService` does not change at all.

## Why DTOs are not here yet

Data Transfer Objects — separate classes for what comes into and goes out of the API — were considered for this MVP. The decision was to leave them out.

DTOs exist at the boundary between the API and the outside world. They control what data gets exposed and what data gets accepted. MVP 1 has no API boundary. There is no controller, no HTTP, no client. Introducing DTOs now would be solving a problem that does not exist yet.

They will arrive in MVP 2b when the API layer is built. At that point the need for them will be obvious and the decision will make sense in context rather than in the abstract.

## What this MVP proves

The class library compiles. The interfaces are defined. The in-memory repository implements the contract. The service calls the repository through the interface.

What it does not yet have is tests. The test project is the next thing — unit tests that prove the repository and service behave correctly before anything else is built on top of them. That is the last piece of MVP 1.
