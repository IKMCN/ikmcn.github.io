With the API running against a real PostgreSQL database, the next step was a way to drive it without opening Swagger for every operation. MVP 2B includes a vanilla JavaScript frontend — a lightweight API console that covers all five endpoints and makes it straightforward to verify the system is working end to end.

The design has one rule: no frameworks, no build tools, no dependencies. A single HTML file you can open directly in a browser.

## Why vanilla JavaScript

The wider project roadmap has the frontend as a swappable layer. The architecture was designed that way from the start — the API does not care what calls it, only that the calls are valid. That being the case, the right tool for a developer console is the simplest one available.

A React or Vue project adds a build step, a node_modules folder, and a local dev server before a single network request has been made. Vanilla JavaScript adds none of those things. A file, a browser, and the API already running — that is the entire setup.

The plan was always to replace this with a more complete frontend later in the series. Starting simple means starting fast, and for the purpose of testing the API there is nothing a framework provides that is actually needed here.

## The structure of the file

The file is divided into three areas: a sticky header containing the base URL configuration, a two-panel layout with the request form on the left and the response on the right, and a sidebar for navigating between endpoints.

All styling is inline using CSS custom properties. All logic is plain JavaScript at the bottom of the file. There is no separation of concerns across multiple files because there is no need for it — the whole thing is a developer tool, not a product.

## Mapping enums to human-readable values

The API represents `CarBodyType`, `CarGearboxType`, and `CarFuelType` as integers. The OpenAPI schema exposes this:

```json
"CarBodyType": {
  "enum": [0, 1, 2, 3, 4, 5, 6, 7],
  "type": "integer"
}
```

A dropdown showing `0`, `1`, `2` is technically correct but useless to work with. The first design decision was to maintain lookup arrays that map each integer position to its label:

```javascript
const GEARBOX   = ['Manual', 'Automatic'];
const BODY_TYPE = ['Saloon', 'Hatchback', 'Estate', 'Coupe', 'Convertible', 'SUV', 'MPV', 'Van'];
const FUEL_TYPE = ['Petrol', 'Diesel', 'Electric', 'Hybrid', 'Hydrogen'];
```

The position in the array is the integer value the API expects. So `BODY_TYPE[2]` is `'Estate'`, which maps to enum value `2`. This is used to populate select elements:

```javascript
${BODY_TYPE.map((b, i) => `<option value="${i}">${b}</option>`).join('')}
```

Each option displays the label but carries the integer as its value. When the form is read, `parseInt(element.value, 10)` gives back the integer the API expects — no extra mapping required.

## Building the car form once, using it twice

The POST and PUT endpoints both need the same set of car fields. Rather than duplicating the HTML for two separate forms, the form is generated by a function:

```javascript
function buildCarForm(prefix) {
  return `
    <div class="field-row">
      <div class="field-group">
        <label>Make</label>
        <input type="text" id="${prefix}-make" placeholder="e.g. Toyota" />
      </div>
      <div class="field-group">
        <label>Model</label>
        <input type="text" id="${prefix}-model" placeholder="e.g. Corolla" />
      </div>
    </div>
    <!-- ... remaining fields -->
  `;
}
```

The `prefix` parameter — `'post'` or `'put'` — is used to namespace the element IDs. This means the two forms can coexist in the DOM without ID conflicts. `post-make` and `put-make` are different elements.

The same prefix is passed to `readCarForm` when it is time to collect the values:

```javascript
function readCarForm(prefix) {
  const str = (id) => document.getElementById(id).value.trim();
  const int = (id) => {
    const v = document.getElementById(id).value;
    return v === '' ? 0 : parseInt(v, 10);
  };
  const num = (id) => {
    const v = document.getElementById(id).value;
    return v === '' ? 0 : parseFloat(v);
  };
  const sel = (id) => parseInt(document.getElementById(id).value, 10);

  return {
    make:       str(`${prefix}-make`),
    model:      str(`${prefix}-model`),
    colour:     str(`${prefix}-colour`),
    year:       int(`${prefix}-year`),
    price:      num(`${prefix}-price`),
    mileage:    int(`${prefix}-mileage`),
    gearbox:    sel(`${prefix}-gearbox`),
    bodyType:   sel(`${prefix}-bodyType`),
    doors:      int(`${prefix}-doors`),
    seats:      int(`${prefix}-seats`),
    engineSize: num(`${prefix}-engineSize`),
    bootSpace:  int(`${prefix}-bootSpace`),
    fuelType:   sel(`${prefix}-fuelType`),
  };
}
```

The helper functions handle the type coercion. An empty number input should not send an empty string to the API — it should send `0`. `parseInt` and `parseFloat` both return `NaN` on an empty string, so the empty check comes first.

The forms are injected into the DOM on load:

```javascript
document.getElementById('car-form-post').innerHTML = buildCarForm('post');
document.getElementById('car-form-put').innerHTML  = buildCarForm('put');
```

## View switching

The five endpoints each have their own view. Only one is visible at a time. The sidebar buttons call `switchView` with the name of the view to show:

```javascript
function switchView(name, btn) {
  views.forEach(v => {
    document.getElementById('view-' + v).classList.toggle('hidden', v !== name);
  });
  document.querySelectorAll('.endpoint-btn').forEach(b => b.classList.remove('active'));
  btn.classList.add('active');
  clearResponse();
}
```

`classList.toggle(className, condition)` takes a boolean as its second argument — `true` adds the class, `false` removes it. Passing `v !== name` means every view except the selected one gets `hidden` applied. The selected one gets it removed. This is neater than separate `add` and `remove` calls in a conditional.

Switching view also clears the response panel. Showing a response from a previous endpoint while looking at a different one would be confusing.

## The fetch abstraction

All five endpoint functions call a single `apiFetch` function:

```javascript
async function apiFetch(method, url, body) {
  const dot = document.getElementById('statusDot');
  dot.className = 'status-dot busy';
  const t0 = Date.now();
  try {
    const opts = {
      method,
      headers: {
        'Content-Type': 'application/json',
        'Accept': 'application/json'
      },
    };
    if (body !== undefined) opts.body = JSON.stringify(body);
    const res  = await fetch(url, opts);
    const ms   = Date.now() - t0;
    const text = await res.text();
    showResponse(method, url, res.status, ms, text);
  } catch (e) {
    showError(method, url, e.message);
  }
}
```

A few things worth noting here.

The status dot is set to `busy` immediately before the request fires, giving visual feedback that something is happening. It transitions to `ok` or `err` once the response arrives, then resets to neutral after three seconds.

Latency is measured by recording `Date.now()` before the fetch and subtracting it after the response arrives. This is client-side latency — it includes network round-trip time, not just server processing time — which is the number that actually matters when testing an API.

The body is only set if something was passed. `GET` and `DELETE` requests do not include a body, so the `if (body !== undefined)` check keeps those requests clean.

The response is read with `res.text()` rather than `res.json()`. This is deliberate. The API returns a plain `true` or `false` for POST, PUT, and DELETE — which is valid JSON but not a JSON object. Reading as text first means the raw value is available regardless of what shape the response takes.

## Displaying the response

Once the response text arrives, `showResponse` handles displaying it:

```javascript
function showResponse(method, url, status, ms, body) {
  let pretty = body;
  try {
    pretty = JSON.stringify(JSON.parse(body), null, 2);
  } catch {}

  lastResponseText = pretty;
  responseBody.innerHTML = syntaxHighlight(pretty);
}
```

Parsing and re-serialising with `JSON.stringify(parsed, null, 2)` pretty-prints the JSON with consistent two-space indentation. If the body is not valid JSON — an empty response, a plain boolean, an error string — the `try/catch` swallows the parse failure and the raw text is used as-is.

The syntax highlighter runs a single regular expression across the formatted text and wraps each token type in a `span` with a CSS class:

```javascript
function syntaxHighlight(json) {
  const esc = escHtml(json);
  return esc.replace(
    /("(\\u[\da-fA-F]{4}|\\[^u]|[^\\"])*"(\s*:)?|\b(true|false|null)\b|-?\d+\.?\d*(?:[eE][+-]?\d+)?)/g,
    (match) => {
      if (/^"/.test(match)) {
        if (/:$/.test(match)) return `<span class="token-key">${match}</span>`;
        return `<span class="token-str">${match}</span>`;
      }
      if (/true|false/.test(match)) return `<span class="token-bool">${match}</span>`;
      if (/null/.test(match))       return `<span class="token-null">${match}</span>`;
      return `<span class="token-num">${match}</span>`;
    }
  );
}
```

The regular expression matches four things: quoted strings (which may or may not be followed by a colon, distinguishing keys from values), the literals `true`, `false`, and `null`, and numbers including decimals and scientific notation. Each match is tested in the callback to determine which span class to apply.

One detail matters here: `escHtml` runs before the regex, not after. This converts `<`, `>`, and `&` to their HTML entities before any spans are injected. Running it after would escape the span tags themselves.

## A note on CORS

Opening an HTML file directly in a browser sends requests with an `Origin` of `null`. By default, ASP.NET Core will reject these requests. During development the API needs a CORS policy that permits this origin:

```csharp
builder.Services.AddCors(options =>
{
    options.AddPolicy("DevPolicy", policy =>
    {
        policy.WithOrigins("null", "http://localhost")
              .AllowAnyHeader()
              .AllowAnyMethod();
    });
});

app.UseCors("DevPolicy");
```

This is development-only configuration. The policy name makes the intent clear — it would not be appropriate to carry this into production.

## What this does not do

The console is not a validation layer. Empty string fields send `""` to the API. Blank number inputs send `0`. There is no UUID format check before a request fires. None of that matters for its purpose — when you are the developer driving the tool you know what valid data looks like.

A more complete client will come later in the series. For now, the goal was a fast, zero-setup way to exercise every endpoint and confirm the system is working. It does that.
